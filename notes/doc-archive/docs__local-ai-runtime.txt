# Local AI runtime (providers, health, and failure contracts)\n
This document describes the local inference and provider path that backs AI APIs and keeps model execution isolated from route handlers.\n

## 2) Execution sequence\n
```mermaid\nsequenceDiagram\n  autonumber\n  participant Route as /api/builder/ai or /api/oracle\n  participant Registry as provider-registry\n  participant Worker as creator-worker / generator\n  participant Manager as model-manager\n  participant Health as model-runtime-health\n  participant Loader as pipeline-loader\n  participant Runner as operation-runner\n  participant Provider as transformers-provider\n  participant DB as knowledge tables\n\n  Route->>Registry: resolve provider\n  Registry->>Manager: execute operation\n  Manager->>Health: gate by circuit/health budget\n  Health-->>Manager: open or openWithWarnings\n  Manager->>Loader: load pipeline lazily\n  Loader->>Runner: execute wrapped operation\n  Runner->>Provider: typed operation run\n  Provider-->>Runner: typed operation result\n  Runner-->>Manager: Result<T>/Failure\n  alt success\n    Manager->>Route: normalized provider result\n  else failure\n    Manager->>Route: failure envelope (retryable + code)\n    Route->>Route: return error envelope\n  end\n  Route->>DB: optionally enrich knowledge context\n```\n
## 3) Failure taxonomy\n\n| Failure code | Semantics | Recovery |\n| --- | --- | --- |\n| `timeout` | budget exceeded before operation completes | retry with backoff if upstream transient |\n| `circuit-open` | repeated failures caused model gate open | wait for cooldown + reset |\n| `cache-corruption-recovered` | pipeline cache mismatch repaired | single auto-recovery then retry |\n| `unavailable` | model/provider disabled or missing config | surface non-retryable unless operator reconfigures |\n| `invalid-output` | malformed provider response | retry once, then report non-retryable with diagnostics |\n| `unexpected` | unknown internal failure | operator triage + optional retry |\n
## 4) Contract guarantees\n
- All local outputs are normalized into internal typed result contracts before route return.\n
- Route handlers never inspect raw transformer output.\n
- Provider health and warmup state are tracked independently from pipeline loading.\n
- Pipeline cache is reused within process with explicit recovery boundaries.\n
## 5) Local knowledge integration\n
- Documents/chunks/terms live under persistent AI tables and are reused by retrieval flows.\n
- Retrieval-assisted generation must include a deterministic context window and explicit truncation strategy.\n
- Search + generation failures stay independent; a retrieval miss should not hard fail generation unless business config requests hard context.\n
## 6) Latency and observability\n
- Each local operation emits timing + model/pipeline identifiers.\n
- Failures are wrapped in a single typed envelope shape and logged with correlation ids.\n
- Health endpoint should expose warmup state, circuit status, and recent failure counts.\n
## 7) 组件演进建议（中文）\n
- 本地模型能力建议保持“单一路径、单一 owner”：\n+  - 路由 -> provider-registry -> model-manager。\n
- 避免在路由里直接拼 prompt 或直接解析模型返回。\n
- 缓存错误应可自恢复：检测到损坏后清理并重建 pipeline。\n
- 健康检查失败与用户可见的生成失败要分离，避免把系统健康问题误报给业务流程。\n*** End Patch
