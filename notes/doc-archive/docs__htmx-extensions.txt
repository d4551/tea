# HTMX lifecycle and extension architecture\n
This document defines ownership and contracts for HTMX enhancement behavior across TEA. It avoids page-local duplication and keeps UX behavior deterministic.\n
## 1) Why centralized extensions exist\n
The architecture keeps progressive enhancement logic in a small set of extensions so builder and game pages share a consistent shell, focus, and validation strategy.\n\n\n+Benefits:\n
- Single implementation for shell busy state and focus restoration.\n+- One path for validation-surface swaps.\n+- Uniform error and toast patterns.\n+- Better accessibility consistency across route boundaries.\n

## 3) Event-to-owner table\n
| Event | Primary owner | Action |\n| --- | --- | --- |\n| `htmx:configRequest` | `layout-controls` | attach request flags, optional CSRF/session context | \n| `htmx:beforeRequest` | `layout-controls` | busy state + request tracing |\n| `htmx:afterRequest` | `layout-controls` | clear busy, restore baseline shell state |\n| `htmx:beforeSwap` | `layout-controls` | allow only approved validation swaps |\n| `htmx:afterSwap` | `layout-controls`/`focus-panel` | rehydrate extension state and focus target |\n| `htmx:afterSettle` | `layout-controls` | final focus and a11y assertions |\n| `htmx:sendError`/`responseError` | `layout-controls`/`server-toast` | clear busy and emit structured feedback |\n| `htmx:load` | `focus-panel` | ensure keyboard focus for swapped panels |\n
## 4) Extension contracts

### 4.1 `layout-controls`\n
- Scope: global shell and route-wide UX state.\n
- Contract:\n  - `busy` tracking per request target.\n  - deterministic `aria-expanded` toggling.\n  - focus restore path after swap.\n  - theme persistence and restore.\n  - `422` validation swap retention.\n
### 4.2 `focus-panel`\n
- Scope: focus ownership for swapped fragments.\n
- Contract:\n  - fragment can declare `[data-focus-panel=\"true\"]`.\n  - focus target must exist before restoring.\n  - fallback target: `#main-content`.\n\n+### 4.3 `oracle-indicator`\n
- Scope: oracle request intent UX only.\n
- Contract:\n  - before request -> explicit loading indicator.\n  - completion -> clear indicator deterministically.\n  - no page-local duplicate indicators.\n
### 4.4 `server-toast`\n
- Scope: toast rendering from server envelope signals.\n
- Contract:\n  - container root fixed by layout.\n  - message localized before UI render.\n  - one renderer, multiple producers.\n\n+### 4.5 `game-hud`\n
- Scope: HUD streaming updates from `/api/game/session/:id/hud`.\n
- Contract:\n  - slot family mapping (scene, dialogue, level, hp/mp/etc).\n  - unknown slots gracefully ignored with logging for diagnostics.\n  - stream data treated as partial overwrite, not merge across unrelated slots.\n\n+## 5) Progressive enhancement policy\n\n- Never add route-local event listeners for busy/focus/validation when shared extension already owns that concern.\n- Avoid duplicate fetch/submit scripts inside templates.\n- Use `hx-*` attributes as the first choice.\n- Route-level scripts should only implement behavior absent from shared extension contracts.\n

## 8) Accessibility requirements\n
- All interactive elements must preserve keyboard order after swaps.\n
- Ensure `aria-live` and toast semantics for transient status.\n
- Do not move focus to hidden/detached nodes.\n
## 9) 中文说明（同步）\n
页面公共行为（加载态、聚焦、验证反馈、Toast）集中在 HTMX 扩展层，避免每个页面重复写脚本。\n
- `layout-controls` 统一处理请求忙态和验证片段。\n
- `focus-panel` 统一处理片段 swap 后的聚焦。\n
- `server-toast` 统一处理服务端消息到用户提示。\n
- HUD 流只通过 `game-hud` 更新，不在业务页面临时重写。\n
