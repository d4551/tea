# Operator runbook (daily operations + incident response)\n
Runbook for bootstrapping, verification, and production-like debugging in this repository.\n
## 1) Environment baseline\n
Required runtime assumptions:\n
- Bun installed and matching policy range.\n
- SQLite database reachable via configured URL.\n
- Asset build tools available and writable output directories.\n
- Optional local AI artifacts available if local providers are enabled.\n
## 2) Golden startup path\n
```mermaid\nflowchart LR\n  S1[bun install]\n  S2[bun run setup]\n  S3[bun run doctor]\n  S4[bun run build:assets]\n  S5[bun run verify]\n  S6[bun run dev]\n\n  S1 --> S2 --> S3 --> S4 --> S5 --> S6\n```\n
## 3) Operational command matrix\n
| Command | Purpose | Expected output |\n| --- | --- | --- |\n| `bun run setup` | first-run bootstrap | env, prisma, assets, readiness checks |\n| `bun run doctor` | structural readiness check | health and filesystem diagnostics |\n| `bun run dev` | local development loop | hot routes, build watchers, realtime |\n| `bun run build:assets` | rebuild CSS/HTMX/client assets | updated public bundles |\n| `bun run docs:check` | docs/manifest validation | no missing archive keys |\n| `bun run lint` | static/type-style checks | no style/type rule failures |\n| `bun run typecheck` | TypeScript compile contract checks | strict boundary checks pass |\n| `bun run verify` | full pre-merge gate | all required gates pass |\n| `bun run dependency:drift` | dependency policy verification | no unexpected version drift |\n| `bun run audit:security` | security audit | dependency risk findings only if present |\n| `bun run start` | production-like boot | startup and route availability |\n
## 4) Verify gate and decision logic\n
```mermaid\nflowchart TD\n  A[bun run verify]\n  A --> B[build:assets]\n  B --> C[dependency:drift]\n  C --> D[docs:check]\n  D --> E[lint]\n  E --> F[typecheck]\n  F --> G[test]\n  G --> H{all pass?}\n  H -->|yes| I[merge allowed]\n  H -->|no| J[fix at source owner]\n  J --> K[re-run verify]\n  K --> A\n```\n
## 5) Incident playbooks\n
### 5.1 Builder publish blocked\n
1. open builder project diagnostics.\n
2. check publish validation output details.\n
3. resolve missing scene/dialogue/asset references.\n
4. re-run publish and confirm snapshot release increments.\n
5. open /game and confirm bootstrap now consumes published snapshot.\n
### 5.2 Game session does not start\n
1. confirm `/game` returned valid bootstrap payload.\n
2. validate project is published and release pointer exists.\n
3. inspect `/api/game/session` call for session + token issuance.\n
4. confirm WS and HUD endpoints can establish with same session id.\n
5. test restore flow with resume token immediately after reconnect.\n
### 5.3 UI state regressions in builder (loading/focus/validation)\n
1. verify `layout-controls.ts` and `focus-panel.ts` extensions are attached.\n
2. check route responses still return envelopes + proper status on 422.\n
3. verify no page-local event listeners duplicate shared extension behavior.\n
4. reproduce with `hx-target` + `hx-swap` path and inspect focus target IDs.\n
### 5.4 AI failures\n
1. check provider registry status via `/api/ai/status` and `/api/ai/health`.\n
2. verify `bun run doctor` and model file availability.\n
3. confirm local model health and circuit state logs.\n
4. if local provider unavailable, switch fallback path by config and retest.\n
### 5.5 dependency or manifest drift\n
1. run `bun run dependency:drift`.\n
2. inspect `bun pm` lockfile mismatch details.\n
3. align package versions with policy and rerun `bun run verify`.\n
## 6) Logging and correlation discipline\n
- correlationId from request context must be present in log lines for major failures.\n
- command, restore, and publish transitions should always log outcome + owner.\n
- avoid ad hoc prints; rely on structured logging helper.\n
## 7) Chinese operator checklist（中文）\n
### 每日流程\n\n
1. `bun run doctor` 检查基础就绪。\n
2. `bun run dev` 做日常开发。\n
3. 每次关键改动后执行 `bun run verify`。\n
4. 发现问题先回到 owning 逻辑（routes/domain/extension）修复，不在前端临时兜底。\n\n
### 关键指标\n\n
- 只允许在可恢复错误（`retryable`）下重试。\n
- 发布前必须通过 docs、lint、typecheck、test。\n
- RMMZ 包文档、主文档索引与档案映射保持一致。\n
5. `bun run start` 进行本地预发自测（可选，发布前).\n
## 8) 一页式排障清单\n
- 先确认文档与契约未漂移（`bun run docs:check`）。\n
- 再确认会话/构建数据一致（`/api/builder/*` 与 `/api/game/*`）。\n
- 再确认构建器发布态和运行态快照关系。\n
- 最后再看容器/依赖层面。\n
