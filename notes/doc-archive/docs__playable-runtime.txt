This is the contract and behavior map for the browser runtime that consumes `/game` bootstrap and game-session transports.

## 3) Transport contract details

### 3.1 Session bootstrap

- Bootstrap JSON is created by game routes and validated in `game-client-bootstrap-session.ts`.
- Required values:
  - `sessionId`, `resumeToken`, `projectId`
  - snapshot build metadata (version + flags)
  - render config (viewport, frame budget, reconnect budget)
  - command defaults (`commandCooldownMs`, `commandTimeoutMs`)

### 3.2 Command loop

- Input layer posts command envelopes to `POST /api/game/session/:id/command`.
- Command responses should be treated as **authoritative outcomes**, not hints.
- Local client may optimistically show “pending” but must not confirm before server response.
- Replay safety:
  - if no response before timeout, switch into reconnect/restore path
  - preserve unsent input queue per frame

### 3.3 Restore and reconnect

```mermaid
flowchart LR
  C1["disconnect"] --> C2["restore attempt with resumeToken"]
  C2 -->|ok| C3["resume state\nre-enter streams"]
  C2 -->|missing token| C4["expired state"]
  C2 -->|failure limit exceeded| C5["need new session"]
  C3 --> C6["UI recovers"]
  C4 --> C7["recreate session CTA"]
  C5 --> C7
```

## 4) Renderer and scene update rules\n
- Renderer owns canvas lifecycle and scene interpolation only.\n
- Renderer reads normalized frame data from transport and updates scene objects deterministically.\n
- Any expensive parsing or schema validation should stay outside the render loop.\n
- Window resize and focus transitions are part of renderer state, not transport state.\n
## 5) State machine for the playable page

```mermaid
stateDiagram-v2
  [*] --> bootstrapMissing
  bootstrapMissing --> bootstrapParseError: parse fails
  bootstrapMissing --> connecting: valid bootstrap\n
  connecting --> connected: WS open + first authoritative frame\n
  connecting --> reconnecting: retryable ws fail\n
  connected --> reconnecting: network drop\n
  connected --> commandPending: user command accepted\n
  commandPending --> connected: command result arrived\n
  reconnecting --> connected: restore success\n
  reconnecting --> restoreExpired: restore failure\n+  reconnecting --> terminalError: repeated non-retryable error\n+  restoreExpired --> terminalError: requires new session\n+  terminalError --> [*]\n+```\n
### State meaning\n
- `bootstrapMissing`: route did not inject payload or script order failed.\n
- `bootstrapParseError`: contract mismatch between server and client.\n
- `reconnecting`: transport-level recovery in progress.\n
- `restoreExpired`: resume token/session pair no longer valid.\n
- `terminalError`: explicit user-facing error; requires action.\n
## 6) Security and anti-cheat boundary\n
- Runtime client never writes authoritative state directly.\n
- Viewer-only participants are filtered by session role before command send.\n
- Session restoration requires server-issued tokens and strict route ownership checks.\n
- Replay or stale frames must be resolved by server envelope + session version, not client clock.\n
## 7) Performance and reliability model\n
- Render loop should stay frame-latency bounded by transport pace.
- Commands are intentionally small and typed to reduce payload and avoid full diff floods.
- Separate channels:
  - WS: full state frames for gameplay continuity
  - HUD SSE: low-cardinality status stream for overlays\n
## 8) Test and verification targets\n
- Command idempotency and schema contract tests under `tests/`.
- WebSocket reconnect + restore tests where transport boundaries are mocked.
- Bootstrap parse tests for locale/render/runtime config changes.
- Replay and spectator restrictions.\n
## 9) 中文说明（同步）\n
可玩运行时强调“服务端权威”，客户端只做命令提交和状态渲染。\n
- 页面加载后先执行 bootstrap 校验，再建立 WS 连接。\n
- `command` 按钮动作以服务端回包为准，不以客户端预测为最终状态。\n
- 恢复流程只在会话/恢复码仍有效时执行，失效后需新建会话。\n
- HUD 与 WS 分离：HUD 用于低频状态，WS 用于主状态流。\n
- 画面恢复、重连、会话过期都映射到单一状态机，避免每页自行处理。\n
