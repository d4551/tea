# Builder domain (complete authoring model and invariants)

The builder domain is the authoritative source of mutable game world data for authoring, and it is intentionally isolated from runtime consumption.

## 1) Primary invariant set

1. **Draft is mutable, published is immutable.**
2. **All builder writes pass through `builder-service`.**
3. **Publish is a gated state transition with validation.**
4. **Runtime seeding consumes only published snapshots.**
5. **Automation and AI patch tools operate within project edit context, never session context.**

```mermaid

  Svc->>Store: persist draft mutation
  Store-->>Svc: saved version\n(optional release counters)
  Svc-->>API: draft update contract

  Author->>API: trigger AI generation/patch
  API->>AI: request plan/generation
  AI->>Worker: run generation job\n(safe, bounded)
  Worker->>Store: persist patch artifact or diagnostics
  Worker-->>Svc: result + risk tags
  Svc-->>API: artifact/patch contract\nfor review or apply

  UI->>API: PATCH /api/builder/projects/:id/publish

    Store-->>Svc: releaseVersion
    Svc-->>API: publish ok + release pointer

    Svc-->>API: validation issues
    API-->>UI: validation fragment + diagnostics
  end
```

## 4) Domain boundaries and owners

| Area | Owner | Responsibility |
| --- | --- | --- |
| Core domain orchestration | `src/domain/builder/builder-service.ts` | create/read/publish/update semantics |
| Draft persistence | `src/domain/builder/builder-project-state-store.ts` | normalize and persist authoring entities |
| Publish gates | `src/domain/builder/builder-publish-validation.ts` | required assets, scene connectivity, NPC/dialogue consistency |
| Automation execution | `src/domain/builder/creator-worker.ts` | scheduled generation job execution |
| Provider selection | `src/domain/ai/provider-registry.ts` | tool generation + retrieval routing |
| Route translation | `src/routes/builder-api.ts` | schema + envelope + view transitions |

  - `validation` for contract mismatches
  - `storage` for persistence issues
  - `authorization` for session-boundary issues
- Publish operations should be safe to retry and idempotent by logical input set.
- All generated patch artifacts should include provenance metadata (`source`, `timestamp`, `traceId`) so failures can be reviewed.
